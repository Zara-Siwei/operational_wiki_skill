# Query Workflow

基于 operational wiki 回答问题。重点是把"概念"和"工具/API"桥接起来。

## 1. 理解问题类型

优先判断属于哪一类：
- 概念问题：`什么是 Debye shielding`
- 操作问题：`PlasmaPy 里怎么计算 Debye length`
- 桥接问题：`教材里的概念在库里怎么落地`
- 比较问题：`书本定义和库文档定义有什么差异`
- 偏好反馈：`以后回答都标注来源`

## 2. 检索顺序

默认顺序：
1. `wiki/index.md`
2. 相关 `concept` 页
3. 相关 `tool` / `api` 页
4. 相关 `source` 页
5. 必要时回 raw 素材

### 桥接问题的优先顺序

如果问题同时涉及物理概念和代码实现：
1. 先读 `concept`
2. 再读 `tool` / `api`
3. 最后用 `source` 页确认证据

## 3. 回答要求

回答时尽量同时给出：
- 概念层解释
- 工具/API 层映射
- 证据来源

对于 `zh` 知识库：
- 回答主体用中文
- 函数名、模块名、包名保留英文

## 4. 自动回写

以下情况可自动更新页面：
- 新发现明确的 concept -> tool/api 联系
- 发现页面缺一条明显 evidence
- 发现 typed relation 缺失
- 发现小错误或拼写问题

## 5. 建议回写

以下情况建议询问用户：
- 需要新建 `analysis`
- 需要新建重要 `api` 页
- 需要对冲突定义做专门比较页

## 6. 偏好回写

若用户给出偏好，写入 `wiki/conventions.md`，不要写到外部 memory。

## 7. 回答格式建议

- 概念问题：定义 + 关键公式 + 来源
- 操作问题：概念解释 + API 路径 + 参数/返回值 + 来源
- 桥接问题：分成"物理意义 / 文档落点 / 代码落点 / 注意事项"
